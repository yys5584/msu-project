# Agent Start Guide

작업 에이전트가 세션 시작 전에 먼저 읽는 공통 기준 문서다.

## 먼저 읽을 순서

1. `AGENTS.md`
2. `README.md`
3. `docs/00_BRIEF.md`
4. `docs/01_GAME_PHILOSOPHY.md`
5. `docs/02_WORKFLOW.md`

## 프로젝트 목적

이 프로젝트는 작은 모바일 게임의 방향, 핵심 루프, 프로토타입 구조를 먼저 명확히 잡고, 이후 대상 제작/배포 환경에 맞춰 출시 가능한 형태로 옮기는 것을 목표로 한다.

가져가야 할 강점:

- 단순한 첫 조작
- 보이는 보상
- 공유하고 싶은 결과
- 신규 유저도 첫날 의미 있는 순간을 경험할 수 있는 구조
- 문서 기반으로 세션이 끊겨도 맥락을 이어가는 방식

## 최상위 판단 기준

1. **보이는 게임인가**
   - 설명보다 화면에서 재미가 먼저 보여야 한다.
   - 결과는 이미지, 모션, 연출로 확인되어야 한다.

2. **자랑할 수 있는가**
   - 유저가 스크린샷이나 짧은 영상으로 공유하고 싶은 장면이 있어야 한다.
   - 결과 화면은 누가, 무엇을, 얼마나 특별하게 얻었는지 즉시 보여야 한다.

3. **신규 유저가 첫날에도 의미 있는 결과를 얻을 수 있는가**
   - 누적 시간만으로 격차가 벌어지는 구조는 피한다.
   - 운, 랜덤 배정, 시즌제, 보정 장치 같은 방법을 적극 검토한다.

4. **첫 조작이 단순한가**
   - 첫 화면에서 해야 할 일이 보여야 한다.
   - 튜토리얼을 읽어야 핵심 루프가 이해되면 실패다.

5. **대상 플랫폼으로 옮길 수 있는가**
   - 프로토타입은 이관을 전제로 작게 만든다.
   - 플랫폼 제약은 추측하지 말고 확인한다.

## 작업 방식

작업 전:

- 문제를 한 문장으로 정리한다.
- 관련 문서를 먼저 읽는다.
- 핵심 루프와 충돌하는지 확인한다.

작업 중:

- 기능보다 먼저 장면을 만든다.
- UI/UX는 UX 구조와 UI 레퍼런스를 분리한다.
- 외부 레퍼런스를 쓰면 출처와 적용 이유를 남긴다.
- 기존에 정의한 역할과 작업 흐름을 재사용한다.

작업 후:

- 변경한 결정사항을 관련 문서에 반영한다.
- 미확정 제약은 추측하지 말고 TODO로 남긴다.
- 다음 작업자가 바로 이어받을 수 있게 파일 기준으로 정리한다.

## 금지

- 랜딩 페이지만 먼저 예쁘게 만들기
- 텍스트 설명으로 보상/결과를 때우기
- 누적 우위만 강화하기
- UI 문제를 예쁜 색과 카드로 덮기
- 플랫폼 제약을 확인하지 않고 단정하기
- 같은 역할과 문서를 매번 새로 만들기
