# UI/UX Planning Rules

## 기본 원칙

UI/UX 작업은 UX와 UI를 분리해서 본다.

- UX: 사용 흐름, 정보 구조, 조작 수, 화면 전환, 긴장/해소 리듬
- UI: 시각 스타일, 레이아웃, 색, 폰트, 컴포넌트, 레퍼런스

## 작업 순서

1. 문제 정의
2. UX 구조안
3. 레퍼런스 기반 UI 방향
4. 화면 초안
5. 제거 전용 검토
6. 시각 QA
7. 구현

## 에이전트 역할

이미 정의한 UIUX 역할을 재사용한다.

### Ideation

역할:

- 넓게 아이디어를 낸다.
- 대안과 변주를 만든다.
- 과감한 방향을 제안한다.

주의:

- 아이디어 제안은 바로 구현하지 않는다.
- 이후 Critique/Remove Only를 반드시 거친다.

### Critique

역할:

- UX 구조가 말이 되는지 본다.
- 정보량, 스압, 조작 수, 혼란을 잡는다.
- 레퍼런스와 실제 문제의 연결성을 검토한다.

### Remove Only

역할:

- 넣을 것을 제안하지 않는다.
- 빼야 할 것, 줄여야 할 것, 기본 노출에서 숨겨야 할 것을 찾는다.
- 랭킹/대시보드/결과 화면처럼 스압 위험이 큰 화면에서 필수다.

## 레퍼런스 사용 규칙

UI 레퍼런스는 가능한 실제 제품/화면 기반으로 찾는다.

기록할 것:

- 검색어 또는 출처
- 선택한 레퍼런스 이름/ID
- 어떤 UI 판단에 적용했는지
- 적용하지 않은 이유가 있으면 그 이유

## 화면 판단 기준

- 첫 화면에서 무엇을 해야 하는지 보이는가?
- 텍스트가 기능 설명으로 과하게 쓰이지 않는가?
- 결과/보상이 한눈에 보이는가?
- 모바일에서 글자와 버튼이 겹치지 않는가?
- 카드/결과/랭킹은 스압 없이 핵심만 보이는가?
- 자세한 정보는 기본 노출이 아니라 확장/상세로 빠져 있는가?

## 금지

- 랜딩 페이지를 먼저 만드는 것
- 설명 카드로 기능을 때우는 것
- 카드 안에 카드 넣기
- 모든 정보를 한 화면에 다 보여주기
- 레퍼런스 없이 감으로 UI 스타일 확정하기
- UX 문제를 예쁜 UI로 덮기
