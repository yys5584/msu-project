# 00. Brief

## 현재 배경

이 프로젝트는 제한된 제작 기간 안에서 작은 모바일 게임의 핵심 루프를 빠르게 검증하고, 이후 대상 플랫폼에 맞춰 출시 가능한 형태로 확장하는 것을 목표로 한다.

대화에서 확인된 주변 맥락:

- 제작 도구와 배포 인프라를 활용할 수 있다.
- 프로젝트는 외부 저장소와 제작 환경을 오가며 관리될 수 있다.
- 심사/성과 기준은 게임 퀄리티와 이용자 유입이 중요할 가능성이 높다.
- 1인 개발자가 3주 안에 MVP를 만들어야 한다.
- 퍼블리싱 지원이 있다고 보고, 개발자는 게임 개발 자체에 집중한다.
- 웹/앱 개발 기반으로 만들고, 모바일 스토어 런칭을 목표로 한다.
- 초기 MVP에서는 UI 디자인 완성도보다 핵심 루프와 반복 재미를 우선한다.

## 목표

대상 제작 환경에서 빠르게 프로토타입을 만들고, 모바일 출시까지 이어갈 수 있는 "보이는 게임"을 만든다.

초기 목표는 대작이 아니라 핵심 장면이다.

## 타겟 유저

1. 짧은 플레이에 반응하는 라이트 유저
2. 이벤트를 통해 유입되는 모바일 유저
3. 결과 공유와 보상 장면에 반응하는 유저
4. 반복 플레이로 즉시 변화를 보고 싶은 유저

## 제품 가설

유저는 복잡한 시스템보다, 짧은 시간 안에 얻는 시각적 결과와 자랑 가능한 순간에 먼저 반응한다.

초기 프로토타입은 아래를 증명해야 한다.

- 짧게 플레이해도 결과가 나온다.
- 결과가 보기 좋다.
- 희귀함이나 변화가 한눈에 보인다.
- 다시 한 번 하고 싶다.
- 남에게 보여주고 싶다.

## 풀어야 할 질문

1. 대상 환경에서 가장 빠르게 보이는 재미를 만들 수 있는 장르는 무엇인가?
2. 모바일 유저에게도 15초 안에 이해되는 장면은 무엇인가?
3. 첫 조작을 얼마나 단순하게 만들 수 있는가?
4. 신규 유저가 첫날부터 자랑할 수 있는 희소성/운/연출은 무엇인가?
5. 커뮤니티에서 공유하고 싶어지는 결과 화면은 무엇인가?
6. 3주 1인 개발로 모바일 출시 후보까지 갈 수 있는 범위인가?
7. 기존 시장 사례와 비교했을 때 직접 중복을 피할 수 있는가?

## 핵심 루프 후보 관리 원칙

구체 후보는 공개 문서에 과도하게 노출하지 않는다. 후보는 내부 검토 문서나 비공개 리서치 결과에서 관리한다.

공개 문서에는 아래 수준만 남긴다.

- 첫 조작
- 첫 결과
- 반복 이유
- 보상/변화가 보이는 방식
- 구현 범위
- 출시 리스크

## MVP 기준

1. 첫 플레이 30초 이내 완료
2. 결과 이미지/화면 1장으로 공유 가능
3. 희귀/대박/실패가 즉시 구분됨
4. 재도전 이유가 명확함
5. 대상 플랫폼으로 옮기기 쉬운 구조
6. 3주 안에 1인이 핵심 루프, 저장, 기본 밸런스, 출시 빌드까지 다룰 수 있음
7. 모바일에서 단순 반복 플레이 가능
8. UI가 미완성이라도 재미 판단이 가능함

## 당장 만들지 않을 것

- 복잡한 장비 강화
- 대규모 오픈월드
- 긴 튜토리얼
- 현금 보상 중심 구조
- 누적형 랭킹만 있는 구조
- 랜딩 페이지만 그럴듯한 프로젝트
- 3주 안에 구현하기 어려운 멀티플레이/소셜 시스템
- UI 완성도에 의존하는 게임성
