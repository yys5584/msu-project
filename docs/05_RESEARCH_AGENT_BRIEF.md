# 05. Research Agent Brief

> 리서치 에이전트를 나누기 위한 공통 브리프. 모든 조사는 3주 1인 MVP, 웹/앱 개발, 모바일 출시 가능성을 기준으로 해석한다.

## 역할

이 문서는 MSU 빌더를 돕는 리서치 에이전트에게 줄 작업 지시서다.

프로젝트 목표와 제약은 `docs/00_BRIEF.md`를 따르고, 리서치 결과로 나온 후보는 `docs/03_IDEATION_BACKLOG.md`에 정리한다.

## 공개 문서 원칙

이 문서는 공개 저장소에 둘 수 있는 수준의 리서치 기준을 담는다.

구체적인 내부 아이디어명, 세부 장르 조합, 민감한 비교 대상은 별도 비공개 문서에서 관리한다.

## 공통 질문

모든 리서치 에이전트는 아래 질문에 답해야 한다.

1. 이 사례에서 유저가 반복한 행동은 무엇인가?
2. 첫 10초에 재미가 보이는가?
3. 30초 안에 결과가 나오는가?
4. 모바일에서 단순 조작으로 가능한가?
5. UI가 거칠어도 재미가 살아남는가?
6. 3주 1인 개발자가 만들 수 있는 핵심만 추리면 무엇인가?
7. 기존 시장 사례와 겹치거나 배울 지점은 무엇인가?

## 리서치 트랙

### Track A. 시장 베이스라인

목표:

- 유사 시장의 장기 강점, 보상 감각, 유저 기대를 정리한다.
- 차용 가능한 감각과 피해야 할 직접 중복을 분리한다.

조사 항목:

- 핵심 재미
- 유저가 좋아하는 보상/연출/수집 요소
- 현재 커뮤니티에서 반복되는 불만
- 짧은 미니게임으로 가져올 수 있는 요소

### Track B. 실패 사례 분석

목표:

- 유사 게임이나 확장작이 왜 유지되지 못했는지 정리한다.
- 장르 전환, 조작, 콘텐츠 구조, 유저 기대 불일치에서 피해야 할 점을 찾는다.

조사 항목:

- 출시 초기 기대와 실제 반응
- 콘텐츠 구조의 문제
- 유저 기대와 어긋난 부분
- 3주 MVP에서 절대 따라 하면 안 되는 범위 확장

### Track C. 성공 사례 분석

목표:

- 최근 성공한 캐주얼/모바일 게임의 구조를 분석한다.
- 직접 중복되지 않게 차별화 포인트를 찾는다.

조사 항목:

- 반복 플레이 구조
- 보상 체감 방식
- 자동화와 직접 조작의 비율
- 유저가 반응한 화면, 숫자, 보상, 캐릭터 요소
- 우리가 피해야 할 직접 중복 요소

### Track D. 유저 제작 게임 분석

목표:

- 유저 제작 게임에서 반복 플레이된 유형을 파악한다.
- 단순 룰, 빠른 반복, 커뮤니티 반응이 있었던 구조를 찾는다.

조사 항목:

- 인기 게임 목록과 장르
- 플레이 시간이 짧은 게임의 공통점
- 유저 제작 게임에서 먹히는 룰
- 재사용 가능한 UX 패턴
- 3주 MVP로 구현 가능한 축약 버전

### Track E. 최근 미니게임 트렌드

목표:

- 최근 글로벌/아시아권 미니게임에서 반응이 좋았던 단순 반복 구조를 찾는다.

조사 항목:

- 앱 내 미니게임 인기 장르
- 짧은 반복 루프
- 짧은 영상으로 반응이 좋았던 장면
- 낮은 UI 완성도에서도 유지되는 재미
- 3주 MVP로 만들 수 있는 최소 버전

### Track F. 커뮤니티 반응

목표:

- 커뮤니티에서 사람들이 실제로 반응한 미니게임/모바일 게임 포인트를 찾는다.
- 조회수나 홍보 문구보다 댓글의 반응 이유를 우선한다.

조사 항목:

- 유저가 중독성 있다고 표현한 이유
- 단순하지만 계속 하게 되는 구조
- 싫어하거나 금방 질린 이유
- 밈/클립/스크린샷으로 퍼진 장면
- 우리 조건에서 재현 가능한 요소

## 산출물 형식

각 에이전트는 결과를 아래 형식으로 낸다.

```md
# Research Result - [트랙명]

## 한 줄 결론

## 주요 출처

- 출처명 / URL / 날짜

## 관찰한 트렌드

## 유저 반응 요약

## 우리 조건에서 만들 수 있는 요소

## 3주 MVP에서 제외할 요소

## 시장 사례 관련 시사점

## 추천 후보

1. 후보명
   - 핵심 조작:
   - 30초 결과:
   - 반복 이유:
   - 리스크:
   - 공개 가능 여부:
```

## 리서치 후 의사결정 기준

리서치가 끝나면 후보를 아래 기준으로 다시 점수화한다.

- 3주 구현 가능성
- 첫 조작 단순성
- 반복성
- 보상 가시성
- 기존 사례와의 차별성
- 모바일 출시 적합성
- UI 완성도 의존도

## 리서치 후 처리

1. 각 트랙 결과를 후보 단위로 쪼갠다.
2. 공개 가능한 후보만 `docs/03_IDEATION_BACKLOG.md`에 옮긴다.
3. 가장 구현 가능한 1개 후보를 `docs/04_PROTOTYPE_DIRECTION.md`에 사양으로 작성한다.
