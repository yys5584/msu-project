# 02. Workflow

## 목적

기획, 구조, 프로토타입, 문서화를 먼저 잡고 대상 제작/배포 환경으로 옮긴다.

초기 작업 환경은 생각과 구조를 빠르게 고도화하는 도구로 사용한다. 대상 제작 환경은 실제 게임 제작/퍼블리싱/인프라 단계의 도구로 사용한다.

## 단계

### 1. 문서 기반 방향 설정

필수 문서:

- `AGENTS.md`
- `docs/00_BRIEF.md`
- `docs/01_GAME_PHILOSOPHY.md`
- `docs/03_IDEATION_BACKLOG.md`

작업 전 질문:

- 지금 만드는 것이 어떤 핵심 장면을 증명하는가?
- 첫 조작과 첫 결과는 무엇인가?
- 유저가 공유하고 싶은 화면은 무엇인가?

### 2. 빠른 프로토타입

먼저 만들 것:

- 핵심 루프의 단일 화면
- 보상/결과 화면
- 짧은 인터랙션
- 데이터 구조 초안
- 게임 밸런스 상수 초안

처음부터 만들지 않을 것:

- 전체 계정 시스템
- 장기 누적 시스템
- 과한 튜토리얼
- 복잡한 상점

### 3. 검증

검증 기준:

- 화면을 캡처했을 때 재미가 보이는가?
- 15초 영상으로 훅이 전달되는가?
- 유저가 무엇을 해야 하는지 보이는가?
- 실패/보통/대박 결과가 명확히 다르게 보이는가?

### 4. 대상 환경 이관

이관 전 준비:

- 핵심 루프 설명 1페이지
- 데이터 구조
- 에셋 목록
- UI 화면 목록
- 애니메이션/연출 목록
- 미해결 제약 목록

확인할 것:

- 지원 런타임/언어
- 에셋 import 방식
- 저장소 연동 방식
- 모바일 빌드/배포 플로우
- 리소스 라이브러리 사용법

### 5. 출시 준비

출시 전 문서:

- 게임 소개 1문단
- 핵심 조작
- 15초 데모 시나리오
- 스크린샷 3장
- 커뮤니티 홍보 문구
- 유저 유입 루프

## 작업 산출물 규칙

- 중요한 결정은 문서에 남긴다.
- 프로토타입은 작게 만든다.
- UI/UX 변경은 `planning/uiux/README.md` 기준으로 남긴다.
- 리서치 결과는 `research/` 아래에 요약한다.
- 플랫폼 제약을 확인한 내용은 이 문서에 갱신한다.
